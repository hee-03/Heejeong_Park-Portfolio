트러블슈팅 & 기술적 의사결정

01
그래프 색상 충돌 — 은행 수 증가 시 동일 색상 반복
원인
Chart.js 기본 팔레트는 7개 이후 색상이 반복되어 은행이 많아지면 구분 불가
해결
HSL 색상 공간에서 은행 수(n)에 따라 360/n 간격으로 색조를 동적 생성하는 유틸 함수 작성
02
권한 우회 시도 — URL 직접 입력으로 타 은행 대시보드 접근
원인
로그인 후 role 기반 페이지 분기만으로는 URL 직접 입력 시 데이터가 노출되는 취약점 발견
해결
백엔드(Flask)에서 세션의 bank_code와 URL 파라미터를 대조. 일반 은행 사용자는 URL 파라미터를 무시하고 세션값으로 강제 고정하여 타 은행 데이터 접근을 원천 차단. 관리자(/admin_dashboard)는 role 검증 후 미충족 시 리다이렉트 처리.
03
빈 데이터 처리 — 신규 은행 온보딩 초기 데이터 없음 상태
원인
이상 거래 데이터가 없는 신규 은행 카드에서 Chart.js가 NaN 오류를 발생시키는 문제
해결
데이터 패칭 후 null/undefined 체크 레이어 추가, 빈 데이터 시 그래프 대신 Empty State 컴포넌트 렌더링

Chart.js로 시간대별 막대그래프·월별 라인차트를 구현하고, onMounted 시 API 호출로 DB 데이터를 실시간 렌더링

채팅 기능 개요

담당 역할채팅 모달 UI 전체 구현 — 채팅방 목록, 실시간 메시지 송수신, 권한별 접근 제어
기술 스택Vanilla JS · WebSocket · CSS · FastAPI · Oracle DB
주요 특징관리자 ↔ 은행 전용 채팅 / 은행 간 직접 채팅 차단 / 모달 기반 UI
관리자와 각 은행 담당자 간 1:1 채팅 기능입니다. 좌측 네비게이션의 채팅 버튼을 누르면 모달로 열리며, 좌측 사용자 목록에서 대화 상대를 선택해 채팅방을 열 수 있습니다. 채팅 UI는 Vanilla JS로 동적 생성되며, 스타일은 별도 CSS 파일로 분리 관리합니다. 은행 사용자 간 직접 채팅은 프론트·백엔드 양측에서 모두 차단합니다.

트러블슈팅 & 기술적 의사결정

01
모달 외부 클릭 시 채팅 상태 유지 문제
원인
모달을 닫았다 재오픈 시 이전에 열었던 채팅방과 메시지 내역이 사라지는 UX 문제 발생
해결
채팅방 메시지를 chatRooms 객체(메모리)에 보존, 모달 display만 토글하여 재오픈 시 상태 그대로 유지
02
오프라인 사용자 메시지 수신 누락
원인
WebSocket은 실시간 연결 기반이라 수신자가 오프라인일 경우 메시지가 전달되지 않는 문제 발생
해결
메시지 전송 시 DB에 IS_READ=0으로 우선 저장, 재접속 시 chat_history API로 과거 메시지 조회하여 복원
03
미확인 메시지 뱃지 — 모달 닫힌 상태에서 새 메시지 감지
원인
모달이 닫혀 있거나 다른 채팅방이 열린 상태에서 수신된 메시지의 읽음 여부를 추적하지 못하는 문제
해결
WebSocket 단일 연결을 페이지 전체에서 유지, 현재 열린 채팅방 외 수신 메시지는 unreadCounts 객체에 누적하여 사이드바·목록 뱃지 실시간 갱신

게시판 기능 개요

담당 역할4개 게시판(공지사항·FAQ·커뮤니티·정밀분석의뢰) 공통 CRUD 구현
기술 스택Flask · Jinja2 · Vanilla JS · Oracle DB
주요 특징동일 UI 구조 재사용 / 게시판별 역할 기반 접근 권한 분리 / 계층형 답글 구조
정밀분석의뢰(1)·커뮤니티(2)·FAQ(3)·공지사항(4) 총 4개의 게시판을 board_type 값 하나로 동일한 템플릿(목록·작성·상세)에서 분기 처리합니다. 각 게시판은 역할(admin/일반 사용자-은행)에 따라 글쓰기·수정·삭제 권한이 다르게 적용되며, 프론트엔드 버튼 제어와 백엔드 Flask 세션 검증을 이중으로 수행합니다.

  • 게시글 클릭 시 상세 화면으로 이동, 상세 진입 시 조회수(readcnt) +1 자동 증가 처리
  • 작성 화면에서 제목·내용·첨부 파일 입력 후 등록, 등록 시 confirm 확인창 표시
