프로젝트 개요

프로젝트명FFG 금융 이상거래 탐지 플랫폼 (FDS Dashboard)
담당 역할프론트엔드 개발 — 관리자 대시보드 · 은행 대시보드
기술 스택React · TypeScript · Chart.js · Axios · JWT · Styled-Components
백엔드FastAPI (Python) · PostgreSQL
협업 도구Git / GitHub · Figma · Notion
금융권 이상 거래를 실시간으로 탐지·시각화하는 플랫폼입니다. 관리자(FFG)는 전체 은행의 이상 지표를 한눈에 모니터링하고, 은행 담당자는 소속 은행의 고객별 이상 거래를 상세 분석할 수 있습니다.

  • 신규 은행 추가 시 별도 수정 없이 동적으로 카드 생성되는 데이터 드리븐 구조 설계

  • API 응답 기반 카드 목록 자동 렌더링 (map 기반 동적 컴포넌트)
  • 점수 임계값(80점 이상) 조건부 스타일링으로 위험 항목 강조

트러블슈팅 & 기술적 의사결정

01
그래프 색상 충돌 — 은행 수 증가 시 동일 색상 반복
원인
Chart.js 기본 팔레트는 7개 이후 색상이 반복되어 은행이 많아지면 구분 불가
해결
HSL 색상 공간에서 은행 수(n)에 따라 360/n 간격으로 색조를 동적 생성하는 유틸 함수 작성
02
권한 우회 시도 — URL 직접 입력으로 타 은행 대시보드 접근
원인
프론트엔드 라우팅 가드만으로는 URL 직접 입력 시 데이터가 노출되는 취약점 발견
해결
JWT의 bank_code와 요청 파라미터를 서버에서 대조 검증. 불일치 시 403 반환하도록 백엔드 팀과 협의
03
빈 데이터 처리 — 신규 은행 온보딩 초기 데이터 없음 상태
원인
이상 거래 데이터가 없는 신규 은행 카드에서 Chart.js가 NaN 오류를 발생시키는 문제
해결
데이터 패칭 후 null/undefined 체크 레이어 추가, 빈 데이터 시 그래프 대신 Empty State 컴포넌트 렌더링

채팅 기능 개요

담당 역할채팅 모달 UI 전체 구현 — 채팅방 목록, 실시간 메시지 송수신, 권한별 접근 제어
기술 스택React · WebSocket · Styled-Components · JWT
주요 특징관리자 ↔ 은행 전용 채팅 / 은행 간 직접 채팅 차단 / 모달 기반 UI
관리자와 각 은행 담당자 간 1:1 채팅 기능입니다. 좌측 네비게이션의 채팅 버튼을 누르면 모달로 열리며, 탭 전환으로 채팅 대상 목록과 채팅방 목록을 구분하여 확인할 수 있습니다. 은행 사용자 간 직접 채팅은 보안 정책상 허용하지 않습니다.

담당 기능 상세

  • 선택된 채팅방의 대화 내역을 시간 순으로 표시, 발신/수신 메시지 좌우 구분
  • 하단 입력창에서 메시지 작성 및 전송, Enter 키 전송 지원

  • 메시지 영역 스크롤: 새 메시지 수신 시 자동으로 최하단으로 스크롤 이동

01
모달 외부 클릭 시 채팅 상태 초기화 문제
원인
모달 닫기 시 컴포넌트가 언마운트되어 선택된 채팅방 상태가 사라지고, 재진입 시 처음부터 탐색해야 하는 UX 문제 발생
해결
채팅 상태(선택된 채팅방 ID, 스크롤 위치)를 전역 상태(Context)로 분리하여 모달 언마운트 후에도 상태 유지
02
WebSocket 다중 연결 — 탭 전환 시 중복 소켓 연결
원인
채팅방 탭 전환 시마다 새 WebSocket 연결이 생성되어 메시지가 중복 수신되는 현상 발생
해결
useEffect cleanup에서 기존 소켓 close 처리, 채팅방 ID를 의존성 배열에 추가하여 채팅방 변경 시에만 재연결
03
미확인 메시지 뱃지 — 다른 화면 이동 중 새 메시지 감지
원인
채팅 모달이 닫혀 있는 동안 수신된 메시지의 읽음 여부를 프론트에서 추적하지 못하는 문제
해결
전역 WebSocket 연결을 앱 최상단에서 유지, 수신 메시지를 전역 상태에 누적하여 모달 미오픈 상태에서도 카운트 갱신

4개 게시판 구현 방식 — 화면별 복사 vs 공통 컴포넌트
고민
게시판마다 개별 페이지를 만들면 코드 중복이 심하고, 공통화하면 예외 케이스 처리가 복잡해지는 트레이드오프 발생
결정
BoardType props 하나로 API 경로·권한·추가 기능(댓글·상태·고정)을 분기하는 공통 컴포넌트 구조 채택. 유지보수성과 확장성 확보
02
상단 고정 게시글 정렬 — 페이지네이션과 충돌
원인
페이지네이션 적용 시 고정 게시글이 2페이지에서도 반복 표시되거나, 1페이지에서 사라지는 정렬 오류 발생
해결
고정 게시글은 별도 API로 항상 상단에 고정 렌더링하고, 일반 게시글 페이지네이션과 분리하여 독립적으로 처리
03
권한 분기 UI — 관리자/은행 사용자별 버튼 노출 제어
원인
게시판마다 권한 조건이 달라(공지: 관리자만 작성, 커뮤니티: 모두 작성 가능) 분기 로직이 컴포넌트 곳곳에 흩어지는 문제
해결
getBoardPermission(boardType, role) 유틸 함수를 만들어 canWrite·canEdit 불리언을 한 곳에서 계산, 컴포넌트에서는 결과만 사용

성과 및 배운 점

🧩
컴포넌트 설계
BoardType props 하나로 4개 게시판을 공통 구조로 처리하는 확장 가능한 설계 경험
🔑
권한 유틸 분리
권한 분기 로직을 유틸 함수로 추출해 컴포넌트 의존성 없이 테스트·재사용 가능하게 구성
📄
UX 일관성
4개 게시판이 동일한 UI 구조를 공유하여 사용자에게 일관된 인터랙션 경험 제공
