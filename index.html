01
그래프 색상 충돌 — 은행 수 증가 시 동일 색상 반복
원인
Chart.js 기본 팔레트는 7개 이후 색상이 반복되어 은행이 많아지면 구분 불가
해결
HSL 색상 공간에서 은행 수(n)에 따라 360/n 간격으로 색조를 동적 생성하는 유틸 함수 작성
02
권한 우회 시도 — URL 직접 입력으로 타 은행 대시보드 접근
원인
프론트엔드 라우팅 가드만으로는 URL 직접 입력 시 데이터가 노출되는 취약점 발견
해결
JWT의 bank_code와 요청 파라미터를 서버에서 대조 검증. 불일치 시 403 반환하도록 백엔드 팀과 협의
03
빈 데이터 처리 — 신규 은행 온보딩 초기 데이터 없음 상태
원인
이상 거래 데이터가 없는 신규 은행 카드에서 Chart.js가 NaN 오류를 발생시키는 문제
해결
데이터 패칭 후 null/undefined 체크 레이어 추가, 빈 데이터 시 그래프 대신 Empty State 컴포넌트 렌더링

채팅 기능 개요

담당 역할채팅 모달 UI 전체 구현 — 채팅방 목록, 실시간 메시지 송수신, 권한별 접근 제어
기술 스택React · WebSocket · Styled-Components · JWT
주요 특징관리자 ↔ 은행 전용 채팅 / 은행 간 직접 채팅 차단 / 모달 기반 UI
관리자와 각 은행 담당자 간 1:1 채팅 기능입니다. 좌측 네비게이션의 채팅 버튼을 누르면 모달로 열리며, 탭 전환으로 채팅 대상 목록과 채팅방 목록을 구분하여 확인할 수 있습니다. 은행 사용자 간 직접 채팅은 보안 정책상 허용하지 않습니다.

모달 외부 클릭 시 채팅 상태 초기화 문제
원인
모달 닫기 시 컴포넌트가 언마운트되어 선택된 채팅방 상태가 사라지고, 재진입 시 처음부터 탐색해야 하는 UX 문제 발생
해결

해결
useEffect cleanup에서 기존 소켓 close 처리, 채팅방 ID를 의존성 배열에 추가하여 채팅방 변경 시에만 재연결
03
미확인 메시지 뱃지 — 다른 화면 이동 중 새 메시지 감지
원인

해결
전역 WebSocket 연결을 앱 최상단에서 유지, 수신 메시지를 전역 상태에 누적하여 모달 미오픈 상태에서도 카운트 갱신

게시판마다 개별 페이지를 만들면 코드 중복이 심하고, 공통화하면 예외 케이스 처리가 복잡해지는 트레이드오프 발생

BoardType props 하나로 API 경로·권한·추가 기능(댓글·상태·고정)을 분기하는 공통 컴포넌트 구조 채택. 유지보수성과 확장성 확보

고정 게시글은 별도 API로 항상 상단에 고정 렌더링하고, 일반 게시글 페이지네이션과 분리하여 독립적으로 처리

AI 분석 개요

담당 역할XGBoost 모델 학습 · SHAP 분석 · 이상 점수 산출 · 대시보드 출력 연동
모델XGBoost (Extreme Gradient Boosting) — 이상 거래 이진 분류

데이터셋Kaggle Credit Card Fraud Detection (mlg-ulb/creditcardfraud)
기술 스택Python · XGBoost · SHAP · pandas · scikit-learn · FastAPI
Kaggle의 신용카드 사기 거래 데이터셋을 활용하여 XGBoost 모델을 학습하고, SHAP으로 각 거래의 피처별 기여도를 분석하여 이상 점수를 산출합니다. 도출된 이상 점수는 FastAPI를 통해 프론트엔드 대시보드로 전달되어 실시간으로 시각화됩니다.

담당 기능 상세

데이터 전처리 & 모델 학습
XGBoost — 신용카드 이상 거래 이진 분류 모델
● 우선순위 상
  • Kaggle Credit Card Fraud 데이터셋 사용 — 284,807건 거래 중 492건(0.17%) 사기 거래, 극심한 클래스 불균형 처리 필요
  • SMOTE 오버샘플링으로 사기 거래 클래스 비율 보정, 모델 편향 방지
  • Amount·Time 피처 StandardScaler 정규화, PCA 변환된 V1~V28 피처 그대로 활용
  • XGBoost 이진 분류 모델 학습, Precision·Recall·F1·AUC-ROC 지표로 성능 평가
기술 포인트
  • 클래스 불균형: scale_pos_weight 파라미터로 정상/사기 비율 자동 반영
  • 하이퍼파라미터 튜닝: max_depth · learning_rate · n_estimators GridSearchCV로 최적값 탐색
  • 과적합 방지: early_stopping_rounds 적용, validation set 기준 조기 종료
SHAP 분석 & 이상 점수 산출
SHAP — 거래별 피처 기여도 분석 및 이상 점수 도출
● 우선순위 상
  • TreeExplainer로 각 거래에 대한 SHAP value 산출 — 모델 예측에 기여한 피처와 기여도를 수치화
  • SHAP value 합산 후 정규화(0~100점)하여 거래별 이상 점수 생성
  • 결제금액(Amount) 기반 이상 점수, 거래위치(V4·V11 등) 기반 이상 점수를 항목별로 분리 산출
  • 항목별 이상 점수와 종합 이상 점수를 FastAPI 응답에 포함하여 대시보드로 전달
기술 포인트
  • shap.TreeExplainer(model) — XGBoost에 최적화된 빠른 SHAP 계산
  • 피처 그룹핑: SHAP value를 결제금액 관련 / 위치 관련 / 기타로 묶어 항목별 점수 분리
  • 이상 점수 정규화: min-max 스케일링으로 0~100 범위로 변환, 임계값 80점 이상을 위험으로 판단

트러블슈팅 & 기술적 의사결정

01
클래스 불균형 — 정상 거래 99.83% vs 사기 거래 0.17%
원인
불균형 데이터 그대로 학습 시 모델이 모든 거래를 정상으로 예측해도 99%+ 정확도가 나오는 문제. Accuracy는 높지만 실제 사기 탐지율(Recall)이 매우 낮음
해결
SMOTE로 사기 거래 오버샘플링 + XGBoost scale_pos_weight 설정. 평가 지표를 Accuracy 대신 Precision·Recall·F1·AUC-ROC로 변경하여 실질적인 탐지 성능 측정
02
SHAP 계산 속도 — 거래 건수 증가 시 응답 지연
원인
거래 건수가 많아질수록 SHAP value 계산 시간이 선형으로 증가하여 API 응답이 수 초 이상 지연되는 문제 발생
해결
단건 계산에서 배치(batch) 처리 방식으로 전환하고, 자주 조회되는 고객의 분석 결과를 캐싱하여 반복 계산 제거. 응답 시간 약 70% 단축
03
이상 점수 해석 — 점수 기준을 어떻게 정의할 것인가
고민
SHAP value는 모델 기여도의 절댓값이라 사용자에게 "이상하다"는 직관적인 의미 전달이 어려움. 어떤 기준으로 0~100점으로 변환할지 결정 필요
결정
전체 거래의 SHAP value 분포를 기준으로 min-max 정규화 적용. 80점 이상을 위험 임계값으로 설정하고, 대시보드에서 해당 항목을 빨간색으로 강조 표시
